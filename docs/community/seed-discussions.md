# Seed Discussions

These are suggested first-wave GitHub Discussions for getting the `Carry` standards conversation moving.

## 1. What Belongs In The Carry Core Profile?

Why this matters:
The fastest way to weaken a standard is to put too much into the core too early.

Starter prompts:
- Which fields are stable enough to belong in the normative profile?
- What should stay in extensions or adapter mappings?
- What would make the core too provider-specific?

## 2. What Should Be The Default Consent Model?

Why this matters:
Portability only works if users trust what is and is not shared.

Starter prompts:
- Should `ask-first` be the default ecosystem posture?
- Which data classes should almost never be auto-shared?
- How should consent travel across providers and devices?

## 3. What Does `Carry Compatible` Mean In V1?

Why this matters:
Compatibility needs to be easy to understand and hard to fake.

Starter prompts:
- Should there be levels such as `import`, `export`, and `round-trip`?
- What is the minimum test bar for claiming compatibility?
- What belongs in fixtures versus in product demos?

## 4. How Should Carry Map Provider Memories From Major AI Systems?

Why this matters:
The standard has to stay neutral while still being practical about provider differences.

Target ecosystems:
- OpenAI
- Anthropic
- Google Gemini
- Meta
- Perplexity

Starter prompts:
- Which provider concepts are common enough to normalize?
- Which provider concepts should stay adapter-specific?
- How do we avoid leaking one provider's model into the core spec?

## 5. What Should AI-Assisted Carry Setup Look Like?

Why this matters:
One of the biggest long-term opportunities is letting an AI help users set up or migrate a `Carry` profile automatically.

Starter prompts:
- What should an assistant be allowed to infer?
- What should always need user confirmation?
- How should AI-generated profile setup interact with consent and provenance?

## 6. How Should Claims And Trust Arrive Without Making Crypto The Product?

Why this matters:
Trustable facts matter, but too much crypto complexity too early can slow adoption badly.

Starter prompts:
- Which facts ever need stronger verification?
- What should stay self-asserted in `v1` and `v2`?
- When should `Carry` align with verifiable credentials more directly?

## 7. What Is The Right Extension Model For Carry?

Why this matters:
The project needs to stay flexible without fragmenting into incompatible dialects.

Starter prompts:
- How should extensions be named and versioned?
- What should the compatibility suite require from extensions?
- How do we keep the core stable while still moving fast?

## 8. When Should Carry Move Toward Shared Governance Or A Foundation?

Why this matters:
Too-early formal governance adds drag. Too-late shared governance can reduce trust.

Starter prompts:
- What adoption threshold should trigger a working group?
- What would justify a foundation or nonprofit?
- Which governance signals should matter more than GitHub stars alone?
